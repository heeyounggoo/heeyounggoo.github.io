# 이력서 임팩트 메트릭 개선 계획

## Context

현재 이력서(`src/data/resumeData.ts`)의 모든 프로젝트가 "작업 나열" 패턴으로 작성되어 있어, 서류 심사자가 성과를 파악하기 어려움. `문제 → 왜 이 방법을 선택했는지 → 측정 가능한 결과` 패턴으로 전면 리라이팅하여 임팩트를 드러내는 것이 목표.

## 수정 대상 파일

- `src/data/resumeData.ts` (유일한 수정 파일)

## 수정 범위

companySummary, details 서술 방식, results 모두 개선

## 프로젝트별 필요 수치 및 수집 방법

### 1. 라이드 - 프론트엔드 개발 환경 설계 (Page 2/6)

| 필요 수치 | 수집 방법 |
|---|---|
| **CI 빌드 시간 (Before/After)** | GitHub Actions 또는 CI 도구에서 모노레포 도입 전후 빌드 히스토리 비교. 캐시 히트 시 vs 미스 시 빌드 시간도 확인 |
| **신규 서비스 셋업 시간** | 모노레포 도입 전 새 프로젝트 생성 시 걸리던 시간(레포 생성, 설정 복사, submodule 연결 등) vs 현재(turborepo generate 등) 비교 |
| **중복 코드 제거량** | git에서 submodule 시절 중복된 파일/모듈 수 vs 현재 공통 패키지로 통합된 모듈 수. 또는 LOC 기준 |
| **온보딩 시간** | 신규 입사자가 개발 환경 셋업 완료까지 걸리는 시간 (체감치라도 OK: "반나절 → 30분") |

**companySummary 개선 방향:**
- 현재: "모노레포 도입과 공통 패키지 구축으로 팀 개발 환경 개선"
- 개선: 핵심 수치 1개 포함 (예: "모노레포 전환으로 CI 빌드 시간 X% 단축, 서비스 간 공유 코드 일원화")

**results 개선 방향:**
- "CI 빌드 시간 단축" → "Turborepo 캐싱으로 CI 빌드 시간 Xm → Ym (Z% 단축)"
- "레포 셋업 간소화" → "신규 서비스 셋업 X시간 → Y분으로 단축"
- "중복 코드 제거" → "web/admin 간 공유 모듈 X개 통합, 중복 코드 약 Y줄 제거"

### 2. 라이드 - 디자인시스템 (Page 3/6)

| 필요 수치 | 수집 방법 |
|---|---|
| **컴포넌트 수** | Storybook에 등록된 컴포넌트 총 개수 확인 |
| **아이콘/SVG 수** | 빌드 스크립트가 처리하는 SVG 파일 수 (디렉토리 내 파일 카운트) |
| **신규 페이지 개발 속도** | 디자인시스템 도입 전후 새 페이지/화면 개발 시 UI 작업에 소요되는 시간 체감치 |
| **디자이너 커뮤니케이션 감소** | 디자인 QA 왕복 횟수 변화, 또는 Storybook 도입 전후 슬랙/회의 빈도 체감치 |
| **번들 사이즈 (Tree-shaking 효과)** | 디자인시스템 패키지 전체 import vs 개별 import 시 번들 사이즈 비교 (webpack-bundle-analyzer 또는 `size-limit`) |

**companySummary 개선 방향:**
- 현재: "일관성 있는 UI와 확장 가능한 컴포넌트 구조를 위한 사내 디자인시스템 설계·배포"
- 개선: 규모감 추가 (예: "X개 공통 컴포넌트와 Y개 아이콘을 포함한 사내 디자인시스템 구축, 신규 화면 UI 개발 시간 Z% 단축")

**results 개선 방향:**
- "커뮤니케이션 비용 감소" → "Storybook 기반 시각적 공유로 디자인 QA 왕복 횟수 X회 → Y회 감소"
- "일괄 반영 구조 확보" → "토큰 기반 테마 시스템으로 전체 서비스 디자인 변경 시 수정 포인트 1곳으로 통일"

### 3. 라이드 - 모빌리티 서비스 개발 (Page 4/6)

| 필요 수치 | 수집 방법 |
|---|---|
| **온라인 계약 전환율 또는 처리 건수** | 서비스 대시보드에서 월간 온라인 계약 건수, 또는 오프라인 대비 온라인 비율 |
| **정책 보완 제안 건수 및 영향** | 기획팀에 제안한 예외 케이스 수, 이를 통해 방지된 데이터 정합성 이슈 |
| **테스트 커버리지** | Jest 커버리지 리포트 (`jest --coverage`) |
| **개발한 화면/기능 수** | 계약 플로우 내 주요 화면 수 (계약서, 서명, 결제, 수납, 서류 등) |

**companySummary 개선 방향:**
- 현재: "오프라인 중심 모빌리티 도메인을 온라인 계약 시스템으로 전환·설계"
- 개선: 규모 또는 비즈니스 임팩트 추가 (예: "월 X건 규모의 자동차 온라인 계약 시스템 설계·개발")

**results 개선 방향:**
- "예외 케이스 발견 → 정책 보완 제안" → "계약 상태 흐름 분석 중 X건의 예외 케이스 발견, 정책 보완 제안으로 데이터 정합성 이슈 사전 차단"
- "Polyfill 구현" → 이건 임팩트가 약하므로, 테스트 커버리지나 다른 수치로 대체 권장

### 4. 핏투게더 - 스포츠 선수 리포트 서비스 (Page 5/6)

| 필요 수치 | 수집 방법 |
|---|---|
| **초기 로드 시간 (Before/After 절대값)** | Chrome DevTools Performance 탭에서 DOMContentLoaded, Load 시간. 현재 "10% 감소"라고만 되어 있는데, **절대값 필요** (예: 3.2s → 2.9s) |
| **빌드 용량 (Before/After 절대값)** | webpack 빌드 결과에서 총 번들 사이즈. "20% 감소"에 절대값 추가 (예: 2.5MB → 2.0MB) |
| **moment→date-fns 사이즈 절감** | moment.js (~300KB) vs date-fns (개별 import 시 ~10-30KB). Bundle Analyzer 캡처 |
| **다국어 지원 언어 수** | i18n에서 지원하는 언어 개수 |
| **번역 키 수** | Google Sheets에서 관리하는 번역 키 총 개수 |
| **번역 업데이트 시간** | 개발자가 직접 하드코딩하던 방식 vs 비즈니스팀이 Sheets에서 수정 후 자동 반영되는 현재 방식의 시간 비교 |

**companySummary 개선 방향:**
- 현재: "초기 로드 성능 개선, 번들 최적화, 다국어 시스템 구축으로 서비스 품질 향상"
- 개선: 핵심 수치 포함 (예: "초기 로드 Xs → Ys 개선, X개 언어 다국어 시스템 구축으로 글로벌 서비스 확장")

**results 개선 방향:**
- "DOMContentLoaded·Load 속도 10% 감소, 빌드 용량 20% 감소" → "초기 로드 Xs → Ys (Z% 개선), 번들 사이즈 XMB → YMB (20% 감소)"
- "i18n 자동화로 생산성 향상" → "X개 언어, Y개 번역 키를 Google Sheets 기반으로 자동화, 번역 반영 시간 X일 → 즉시"
- "위젯 간 데이터 충돌 해소" → 이건 이미 구체적이므로 유지

### 5. ANTLabs - 대학교 학사 시스템 (Page 6/6)

| 필요 수치 | 수집 방법 |
|---|---|
| **전환한 화면/모듈 수** | Flex에서 SPA로 전환한 주요 화면 개수 |
| **페이지 전환 속도** | Flex 전체 리로드 vs SPA 라우팅. Flex는 통상 2-5초, SPA는 수백ms |
| **공통 컴포넌트 재사용 횟수** | Editor, FileUploader, TimeTable 등이 교원 시스템 내 몇 개 화면에서 사용되는지 |

**companySummary 개선 방향:**
- 현재: "Flex 레거시 시스템을 Vue.js SPA로 전환, 공통 컴포넌트 설계"
- 개선: 규모 추가 (예: "X개 화면의 Flex 레거시를 Vue.js SPA로 전환, 공통 컴포넌트 X종 설계")

**results 개선 방향:**
- "페이지 전환 속도 및 사용성 개선" → "SPA 전환으로 페이지 전환 속도 Xs → Xms로 개선"
- "반복 개발 제거" → "공통 컴포넌트 X종을 Y개 화면에서 재사용, 반복 개발 제거"

## 수집이 어려운 경우 대안

과거 프로젝트(핏투게더, ANTLabs)는 정확한 수치 확인이 어려울 수 있음. 이 경우:

1. **기억 기반 추정 + "약" 접두사**: "약 40% 개선", "약 2배 단축"
2. **상대적 표현**: "Xs에서 Ys로", "반나절 → 30분"
3. **규모 표현**: "X개 화면", "X종 컴포넌트", "X개 언어"
4. **빈도/횟수**: "주 X회 → Y회", "월 X건"

숫자가 하나도 없는 것보다 추정치라도 있는 것이 훨씬 나음. 다만 과장은 면접에서 역효과.

## 구현 방식

1. 사용자가 위 표의 수치를 확인/결정
2. `src/data/resumeData.ts`에서 다음을 수정:
   - 각 프로젝트의 `companySummary` 리라이팅
   - `details` 내 서술을 "왜 이 방법?" 맥락 추가
   - `results` 배열을 수치 포함 임팩트 문장으로 교체
3. em dash 사용 금지 (CLAUDE.md 규칙)

## 검증

- `npm run build`로 빌드 에러 없는지 확인
- `npm run dev`로 이력서 페이지에서 텍스트가 정상 렌더링되는지 확인
- 인쇄 미리보기(Ctrl+P)로 6페이지 레이아웃이 깨지지 않는지 확인 (텍스트 길이 변경으로 페이지 넘침 가능)
